## 一、先说说那些让我们崩溃的日常

### 崩溃现场一：验证同事的灵魂拷问

流片前两周，验证大佬拿着一张泛黄的时钟树图来找你：

> "这个 `axi_clk` 的源头不是 `pll_mux_clk` 吗？为什么我仿真抓出来是 `pad_src_clk`？"

你打开 RTL，沉默三秒。

三个月前架构改了，MUX 默认选择从 PLL 切到了 Pad。Excel 表格更新了——毕竟那是规范，不敢不改。但那张 draw.io 的拓扑图？**早就不知道烂在哪个版本的文件夹里了。**

你只能说："图是旧的，以代码为准。"

验证大佬点点头，走了。但你心里清楚：**以代码为准？那我要这张图干嘛？**

### 崩溃现场二：三个人，三个版本

芯片项目里永远存在神秘的三重宇宙：

- **RTL 代码**（前端工程师写的，昨天刚改了一版）
- **Excel 规格表**（架构师维护的，上周更新过）
- **draw.io 拓扑图**（项目 kickoff 时某位热心同事画完就再也没人管了）

然后验证同事就会问出那个经典问题：

> "代码里写的是 `pll_mux_clk`，Excel 里写的是 `pad_src_clk`，图里画的是 `div_src_clk`……**大哥，我到底该信哪个？**"

前端工程师："代码肯定是对的，我亲手写的，不信你看 commit log。"

架构师："Excel 是最新的，代码可能没合入最新版本——当然也可能合了但我没更新 Excel，这我得查一下。"

你："图是照着老版 Excel 画的，后来 Excel 改了，图没更新。"

**三个人说得都不一样，但每个人都觉得自己是对的。**

最绝的是什么？是如果代码和图**连错的都一样**，你根本发现不了问题——因为看起来它们"完全对齐"。只有当验证抓出异常、或者流片后功能不对，你才发现某个中间版本早就被所有人遗忘了。

**bug 藏得最深的时候，不是代码和图不一样，而是它们错得整整齐齐。**

### 崩溃现场三：验证阶段，bug 单像雪花一样飞来

流片前最忙的不是前端，是验证。

验证同事吭哧吭哧跑了两个月回归，然后抱着一摞 bug 单来找你：

> "提了 47 个单，你猜怎么着？**32 个都是图和代码对不上。**"
>
> "这个时钟源头和拓扑图标的不一样。"
>
> "分频比文档写的是 DIV2，代码里是 DIV4。"
>
> "这个 ICG 在图里有，RTL 里压根没找到。"
>
> "复位路径和规格表描述的完全不一致。"

你一张张核对，越核对越绝望：

- 有些是因为代码改了，图没更新
- 有些是因为架构调整了，Excel 没同步
- 有些是因为图里画错了，代码其实是对的
- 最离谱的是：**有 5 个单，图和代码明明写的是一样的，但仿真波形显示都不对**——说明某个中间版本被所有人遗忘了，代码和图一起错了

你花了一整天，把这 32 个单逐一核对、解释、关闭。

验证同事问："那真正的设计 bug 有几个？"

你说："……5 个。"

**80% 的精力耗在'文档对账'上，只有 20% 在真正找设计问题。**

更惨的是，验证Leader向项目经理汇报的时候说："这模块验证效率低，bug 特别多。"项目经理看向前端总监，前端总监看向你。

你很想解释："不是设计 bug 多，是图纸太烂了。"但你张了张嘴，什么也没说。

**因为图纸烂，本来就不是你的 KPI。但它害你背了锅。**

### 崩溃现场四：CRG 需求变了八次，图还是第一版

这是最隐形、最频繁的崩溃。

芯片设计周期里，CRG（Clock Reset Generator）是**变动最频繁的模块**，没有之一：

- **第一周**：架构定好了，`core_clk` 跑 1000MHz
- **第三周**：老板说功耗太高，`core_clk` 降到 800MHz，分频从 DIV2 改成 DIV4
- **第五周**：验证发现某些场景需要测试时钟，加个 MUX 切换
- **第七周**：后端说 skew 太大，某些路径要加 ICG 门控
- **第九周**：软件说某个模块可以常驻关闭，再加一个 OCC 控制点
- **第十一周**：DFT 工程师过来说测试时钟走不到某个角落，要改 MUX 位置

**每变一次，Excel 得改，RTL 得改，图……算了先不改，等最终版再画。**

然后你就有了：
- **Excel v3.2**（最新）
- **RTL v2.8**（上周合入的）
- **拓扑图 v1.0**（项目 kickoff 时画的，再也没更新过）

**CRG 设计需求不停的变化，时钟频率变、默认分频变、多个时钟、少个时钟，而图永远停留在最初版本。这就是图码不一致的终极根源。**

### 崩溃现场五：画一张图，一下午没了

老板周一上午说："这周评审会，你把时钟树图画一下。"

你打开 draw.io，开始：

- 拖一个方块，写 `pll_src_clk`
- 拖一个方块，写 `pll_mux_clk`
- 画一条线……不对，线怎么歪了？
- 调一下对齐……再调一下……
- 这条线要跨过去，怎么画跳线？arc 还是 gap？
- 这个颜色配得不好看，换个绿的
- 等等，MUX 的输入顺序反了，SRC0 应该在上面
- 这个节点的字太多，框太小了，拉大一点
- 怎么又有两条线交叉了？
- 这个时钟叫 `axi_aclk` 还是 `axi_clk`？翻一下 Excel……哦是 `axi_clk`，但图里已经写成 `axi_aclk` 了
- 漏了一个 `sram_clk`，从哪接过来的？让我数一下……第 17 个信号，从 `pll_mux_clk` 出来经过 DIV2……
- 等等，刚才那个 DIV 的分频比写的是 2 还是 4？让我再翻一下 spec……
- ……

**一下午过去了，你画了 8 个节点，还有 40 个没画。而且你已经忘了前面 8 个有没有写错名字、连错线、标错分频比。**

手动画图的痛苦不只是慢，更在于：
- **连线**：一条一条手动画，几十条线画完眼睛都花了
- **跨线编辑**：线多了交叉，调跳线、调 arc、调 gap，每一根都要手动处理
- **写文字**：每个节点手写名称，写错一个字母就是误导
- **容易遗漏**：Excel 里有 30 个时钟，图里只画了 28 个，漏了哪两个？不知道
- **笔误**：`axi_clk` 写成 `axi_aclk`，`1000MHz` 写成 `100MHz`，自己检查不出来

更崩溃的是，周三架构师说："哦对了，那个 `core_clk` 的分频从 DIV2 改成 DIV4 了，你图里改一下。"

你看着那张好不容易对齐的图，内心只有一个念头：**我当初为什么要学芯片设计？**

## 三、这东西到底怎么做到的？

它叫 **CR Tree Diagram Generator**，本质上是一个 **AI Skill**。

什么是 AI Skill？你可以把它理解为**给 AI 写了一份"上岗培训手册"**。

### 第一份文件：SKILL.md —— AI 的"培训手册"

这份文件不是给你看的，是给 AI 看的。里面写的是：

- **你能干啥**：生成时钟树/复位树拓扑图
- **什么时候出手**：用户说"画时钟树""生成拓扑图""可视化时钟"的时候
- **要啥材料**：一个 Excel 表格
- **出啥成果**：Draw.io + Excalidraw 双格式
- **具体步骤**：第一步读表、第二步建链路、第三步出图

**AI 看了这份手册，就知道自己什么时候该干活、怎么干活。**

### 第二份文件：MCP Server —— AI 的"手脚"

SKILL.md 告诉 AI "该做什么"，MCP Server 告诉 AI "怎么做"。

MCP（Model Context Protocol）是个标准协议，让 AI 能调用外部工具。就像你给实习生配了一台电脑，SKILL.md 是培训手册，MCP Server 就是电脑里的软件。

当用户对 AI 说"帮我画个时钟树"时：

1. AI 翻开 SKILL.md，心想："这活儿我能接"
2. AI 通过 MCP Server 调用工具
3. 工具自动解析 Excel、生成拓扑图、保存文件
4. AI 把结果丢给用户

**你看到的是 AI 十秒出图，你看不到的是背后 SKILL.md + MCP Server 的完美配合。**

## 五、时钟树到底是怎么一回事？

在聊好处之前，先快速科普一下：时钟树没有你想的那么神秘。

它本质上就是**几个基础组件像搭乐高一样拼起来**。

### 组件一：MUX（多路选择器）

两个时钟进，一个时钟出。SRC0 和 SRC1 是两个输入，SEL 是选择信号。

```
     pll_src_clk ──┐
                   ├──[ MUX ]──→ pll_mux_clk
     pad_src_clk ──┘
```

用处：系统上电时用外部时钟，PLL 锁定后切换到内部时钟。或者测试模式切到测试时钟。

### 组件二：DIV（分频器）

时钟进去，频率降一半（或 1/4、1/8）出来。

```
     pll_mux_clk ──[ DIV2 ]──→ core_clk (1000MHz)
```

1000MHz 进来，500MHz 出去。简单直接。

### 组件三：OCC（时钟控制器）

控制时钟什么时候开、什么时候关。通常和软件寄存器联动。

```
     div_clk ──[ OCC ]──→ 受控时钟
                 ↑
             sw_ctrl (软件控制)
```

用处：某个模块暂时不用了，OCC 帮你把时钟关掉，省电。

### 组件四：ICG（时钟门控）

OCC 的"执行器"。OCC 说"关"，ICG 就把时钟掐断；OCC 说"开"，ICG 就放过去。

```
     occ_clk ──[ ICG ]──→ gated_clk
                ↑
            enable
```

### 拼起来就是一棵时钟树

```
     pll_src_clk ──┐
                   ├──[ MUX ]──→ pll_mux_clk ──[ DIV2 ]──→ occ_clk ──[ ICG ]──→ core_clk
     pad_src_clk ──┘
```

**Source → MUX → DIV → OCC → ICG → Output**

这就是时钟树的全部秘密。没有什么黑魔法，就是这几个组件按不同的顺序、不同的参数组合起来，形成一棵从 PLL 到各个模块的"树"。

---

### 那 DFT 和 OCC 又是什么鬼？

上面讲的是**功能模式**下的时钟树。但芯片造出来之后，还得测它能不能正常工作。这就涉及到 **DFT（Design For Test，可测性设计）**。

**问题**：芯片在 ATE（自动测试设备）上测试时，功能时钟可能跑得太快（比如 1GHz），测试设备跟不上。怎么办？

**解决办法**：在时钟树里放一个叫 **OCC（On-Chip Clock，片上时钟控制器）** 的模块。它的本质是一个 MUX：

- **正常模式**：选功能时钟（PLL 输出的高频时钟）
- **测试模式**：选测试时钟（外部输入的低频时钟，比如 50MHz）

```
       pll_clk ──┐
                 ├──[ OCC (MUX) ]──→ testable_clk
    test_clk ──┘        ↑
                    test_mode
```

DFT 工程师最关心的问题：
- 测试时钟能不能走到**每一个寄存器**？
- OCC 点放在哪里最合适？太靠近根节点，测试覆盖不全；太靠近叶子节点，MUX 太多浪费面积
- 扫描链（Scan Chain）上的寄存器，时钟路径有没有被门控掐断？

**所以 DFT 工程师特别需要一张准确的时钟树拓扑图**。图上 OCC 点在哪里、测试时钟怎么走的、哪些路径有 ICG 门控——一目了然。

以前 DFT 工程师要手动追踪 RTL 里的时钟路径，或者拿着一张过时的图猜来猜去。现在 Excel 更新，图自动更新，DFT 和前端看到的永远是同一张图。

### 复位树：多个复位源头的"与"逻辑

复位树比时钟树更朴素。它的核心思想是：**任何一个复位源有效，最终输出就应该复位。**

芯片里通常有多个复位源：
- **POR**（Power-On Reset）：上电复位，芯片刚上电时自动产生
- **Soft Reset**：软件复位，通过寄存器配置触发
- **WDT Reset**：看门狗复位，系统跑飞时自动触发
- **External Reset**：外部引脚复位

这些复位信号通过**异步与逻辑**（Async AND）组合到一起：

```
      por_rst_n ──┐
    soft_rst_n ──┤
     wdt_rst_n ──┼──[ Async AND ]──→ core_rst_n
       ext_rst ──┘
```

**为什么是"与"逻辑？** 因为复位信号通常是低电平有效（`_n` 后缀）。只要任意一个复位源拉低（有效），AND 门的输出就拉低，目标模块就被复位。

它和时钟树的区别：
- 时钟树是**传播**——把时钟从 PLL 分发到各个模块
- 复位树是**汇聚**——把多个复位源合并成一个统一的复位信号

但无论时钟树还是复位树，本质上都一样：**几个基础组件按不同的拓扑连接起来，形成一棵从源到目标的链路。**

### 等等，实际的时钟树比这复杂多了？

没错。上面那个是教科书式简化版。真实芯片里的时钟树可能是这样的：

```
     pll_src_clk ──┐
                   ├──[ MUX ]──→ pll_mux_clk ──[ DIV2 ]──→ mid_clk ──[ MUX ]──→ branch_clk ──[ DIV4 ]──→ occ_clk ──[ ICG ]──→ final_clk
     pad_src_clk ──┘                                           ↑
                                                      test_clk ──┘
```

两个 MUX 级联、两个 DIV 级联、中间还夹着一个 MUX。**深度任意长，组件顺序任意排。**

这个 Skill 不限制你的时钟树必须按"MUX → DIV → OCC → ICG"的标准顺序来。Excel 里怎么写，它就怎么画。你的链路是 3 层还是 10 层，它都吃得下。

## 六、这个东西到底好在哪？

### 好处一：Excel 就是唯一真相源

你的时钟配置本来就是 Excel 表格维护的——那个表通常就是设计规范，甚至可以直接从寄存器配置导出。

**以前**：改 RTL → 更新 Excel → 手动打开 draw.io 改图 → 改到一半被拉去开会 → 回来忘了改到哪了 → 图烂了

**现在**：改 RTL → 更新 Excel → 对 AI 说"重新生成一下" → 10 秒出图

**从半小时到 10 秒，效率提高 1000%。**

这意味着什么？

以前 CRG 改一版，你一想到还要重新画图，本能地拖延——"等最终版再画吧"。结果一拖再拖，图永远停留在 v1.0。

**现在需求变了，立马改 Excel，对 AI 说句话，10 秒后新图就出来了。一天迭代 10 次都不怕。**

早上架构师说分频从 DIV2 改 DIV4，你改完 Excel，10 秒出新图。下午 DFT 说 MUX 要换个输入顺序，你再改 Excel，10 秒又出新图。晚上后端说某条路径要加 ICG，你再改——还是 10 秒。

**每一次代码变更，图同步更新。代码和图一起发布，永远对齐。**

而且因为每次都是从 Excel 重新生成，**图永远是 Excel 的实时镜像**。不存在"图是图、代码是代码"的割裂。

**表格 = 唯一真相源。图 = 自动衍生产物。**

### 好处二：双格式输出，一次搞定两种场合

- **Draw.io 格式**：方方正正、配色规范、连线整齐。适合放进设计文档、评审 PPT、技术规格书——**看起来就很专业，老板看了点头，客户看了放心。**
- **Excalidraw 格式**：手绘风格、线条抖动、像人用笔在纸上画的。适合发团队群、发朋友圈、发公众号——**看起来就不像机器生成的，特别有亲和力，一眼就能抓住注意力。**

**同样是时钟树图，正式版和手绘版给人的感觉完全不一样：**

- 正式版像西装革履的架构师站在白板前讲方案——严谨、可信、适合归档
- 手绘版像同事在餐巾纸上随手画的草图——亲切、易懂、适合讨论

你画了一下午 draw.io，发群里没人回。你随手丢一张 Excalidraw 手绘图，群里瞬间刷屏"这个好看""求原图""怎么画的"。

**因为手绘风格自带一种"这个人真的懂，而且愿意花时间画给我看"的错觉——虽然其实是 AI 十秒生成的。**

一份数据，两种表达。给老板看正式的，给同事看手绘的，发朋友圈也用手绘的。**见人说人话，见鬼说鬼话，见朋友圈发手绘的。**

### 好处三：一张图，全团队用，随版本走

这张时钟树图不是给你一个人看的。

| 角色 | 用这张图干什么 |
|------|---------------|
| **前端设计** | 确认时钟链路实现和规格一致，检查有没有漏接 |
| **验证** | 追时钟源头、debug 时序违例、确认门控策略 |
| **软件** | 了解有哪些可配时钟、分频比、门控寄存器 |
| **DFT** | 检查 OCC 点位置、确认测试时钟可达性、做 scan 插入 |
| **中端/后端** | 做时钟树综合（CTS）、分析 skew、优化绕线 |

以前每个人手里拿的可能都是不同版本的图：
- 前端改了一版，验证手里的还是上个月的
- DFT 画了一张测试时钟图，后端看的又是另一张
- 软件工程师根本不知道哪个时钟可以被门控

**现在 Excel 是唯一的真相源，图是自动生成的。RTL 版本更新 → Excel 更新 → 图自动更新。随版本发布，全团队看到的永远是同一张图。**

没有"你手里的图是 v1.2，我手里的是 v1.3"这种尴尬了。

### 好处四：复杂拓扑它都懂，DFT 也考虑了

不只是简单的"源 → 输出"，芯片设计里那些乱七八糟的结构它都能自动处理：

- **MUX 选择器**：SRC0/SRC1 双输入，自动画成梯形
- **分频器 DIV**：自动插入并标上分频比，支持多级级联
- **OCC/ICG**：时钟门控控制器自动识别
- **OCC 点支持**：支持 DFT 测试时钟切换（On-Chip Clock），ATE 测试模式下自动切到测试时钟源
- **任意排列组合**：MUX 可以在 DIV 前，也可以在 DIV 后；可以有多个 MUX 串联；链路深度不限
- **复位树**：SOFT/REG/AND 节点统统安排

**DFT 工程师特别在意这个**：测试时钟能不能走到每个寄存器？OCC 点放的位置对不对？以前要一张张手动检查，现在图上一眼就能看出来测试时钟的完整路径。

**你不需要告诉它'这里有个 MUX'，你只要 Excel 里写了 SRC0/SRC1，它自己就懂了。你的时钟树是 3 层还是 10 层，它都照单全收。**

### 好处五：频率直接标在图上

Excel 里的 `NOTE` 列（比如 `1000MHz`、`200MHz`），自动显示在节点框里。

老板看图就能知道每个时钟跑多快，不需要再翻表格。当然，如果频率标错了，那也是 Excel 的锅，不是图的锅。**甩锅变得更有理有据了。**

### 好处六：不同来源的边用不同颜色

来自 PLL1 的链路用红色，来自 PLL2 的用蓝色，来自外部 Pad 的用绿色……

一眼就能看出哪个输出属于哪个时钟域。这对于那种"我明明关了 PLL1 为什么还有时钟在跑"的 debug 场景，简直是救命稻草。

## 七、安装？五分钟搞定

### 第一步：下载

```bash
git clone https://github.com/siliconpeasant/soc_agent_skill.git
```

### 第二步：装依赖

```bash
pip install pandas openpyxl
```

### 第三步：配置 MCP

在 `~/.kimi/mcp.json` 里加一段：

```json
{
  "mcpServers": {
    "cr-tree-diag-gen": {
      "command": "python3",
      "args": ["/path/to/cr_tree_diag_gen/mcp_server.py"]
    }
  }
}
```

重启 Kimi，完事。

**从此以后，你再也不用记任何命令。说话就行。**
